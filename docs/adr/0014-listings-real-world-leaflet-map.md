# 0014 — Listings map: real-world Leaflet satellite map replaces the stylised SVG

- **Status:** accepted
- **Date:** 2026-07-03
- **Supersedes:** ADR 0005 (custom SVG region map)
- **Relates to:** 0010 (three-tier location), 0011 (map Cluster tier), 0013 (zoning tool's Leaflet map)

## Context

ADR 0005 built the Property & Land Explorer's map as a hand-drawn SVG because
listings carry no coordinates and a map library added weight for no pins. Since
then the Zoning & Land Check tool (ADR 0013) shipped its own Leaflet satellite
map, so Leaflet 1.9.4 + its CSS are already loaded globally on every page, the
SPA init pitfalls (deferred init, `fadeAnimation:false`, ResizeObserver, settle
watchdog, base.css reset override) are solved and documented, and the Esri
World Imagery tiles are keyless and free. Jon asked for the listings map to
become a real-world map like the zoning one.

Listings still have **no coordinates** (location is categorical: Region → Area
→ Place keys — see docs/listings-location-model.md), so per-listing pins remain
impossible; the map stays a **location filter**.

## Decision

Rewrite `createLombokMap()` in app.js as a **Leaflet map** (Esri World Imagery
satellite, keyless, same tiles as zoning) with the **identical component
contract** — `createLombokMap(wrapEl, {onSelect})` returning
`{setCounts, relayout, setSelection}` — so the listings filter engine, URL
state, back button, and mobile collapsible panel are untouched.

- **Geometry is reused, not redrawn.** The 880×640 viewBox paths in
  `LOMBOK_MAP` (real OSM coast + curated market-region splits, generated by
  `docs/tools/build_lombok_map.py`) are **unprojected back to lat/lng at
  runtime** via `LOMBOK_MAP.geo` — the exact inverse of the build script's
  equirectangular projection (`lon = lon0 + x/lonScale`,
  `lat = lat0 - y/latScale`), recovered by least-squares fit against the baked
  markers (max residual ≈ 5 m). Region polygons, cluster zones, bboxes and
  zoom boxes all convert through it, so the market Regions land on the real
  island and `build_lombok_map.py` stays the single geometry origin.
- **Markers get true coordinates.** Area/Place markers use the real lat/lng
  from the build script's `AREAS`/`PLACES` tables, baked into
  `LOMBOK_MAP.geo.areas/places` — NOT the `p:[x,y]` values, which were
  coast-snapped for the stylised look. Positions are Jon's curated best-effort;
  refine by editing `LOMBOK_MAP.geo` (keep the script in sync).
- **Interaction parity.** Same drill levels (island → Region → Cluster → Area →
  Place), same click semantics (region toggle, cluster zones with summed
  counts, marker-*name* click commits + scrolls on mobile while dot click
  refines in place), same live counts and empty-state dimming/dead zones, two-way
  sync with the dropdowns. Selection changes `flyToBounds`; free pan/zoom is
  allowed and never fought (re-fit only on real selection change or relayout).
- **Rendering:** region fills are translucent warm-cream polygons with thin
  light strokes over satellite; labels are `divIcon`s (tracked uppercase,
  text-shadow) styled by `.lmapl-*` classes in style.css. Place-name labels
  show at cluster level or zoom ≥ 13.5 (`.lmapl-deep`) with the old
  diagonal-lane treatment at cluster zoom. Zoom control sits top-right (top-left belongs to the back
  button). No Esri reference-labels layer for now — our own labels carry the
  info (revisit if orientation suffers).
- **Hardening** copies zoning.js: deferred init until the SPA attaches the
  container, `fadeAnimation:false`, ResizeObserver + ~7 s size-settle watchdog,
  and the base.css `img/svg/canvas` reset neutralised inside `.lmapl`.

## Why this and not the alternatives

Keeping the SVG (status quo) was rejected by the owner: a real-world satellite
map is what buyers expect and matches the zoning tool. A from-scratch marker
config was rejected because the projection inverse lets ~700 lines of owned,
already-calibrated geometry (region boundaries no GeoJSON source can provide —
they are *market* regions) transfer losslessly. Google Maps was rejected for
API billing and brand reasons (unchanged from ADR 0005). Waiting for listing
coordinates was rejected — ADR 0005 itself said to revisit the map decision
rather than bolt pins onto the SVG; this ADR is that revisit, and pins can now
be added incrementally once coordinates exist.

## Consequences

- `LOMBOK_MAP.outline`/region `d` paths remain load-bearing (toggle icon +
  unprojected polygons). The terrain WebP (`images/lombok-terrain.webp`) is no
  longer rendered; the `--lmap-u` SVG zoom-compensation trick and most
  `--lmap-*` theme vars are gone (the few left style panel chrome).
- If `build_lombok_map.py` is ever re-run with a different bounds fit, the
  `LOMBOK_MAP.geo` constants must be re-derived or the polygons will shift.
- The listings map now needs network tiles (Esri). Attribution is shown on-map;
  the map degrades to a text fallback if Leaflet is absent.
- ADR 0011's Cluster tier carries over unchanged (still view-only config).
- **Per-listing pins are now one schema step away**: capture
  `product.geo` GeoCoordinates in the worker extractors (currently discarded),
  add `listings.latitude/longitude`, and drop pins on this map — a natural
  follow-up, deliberately out of scope here.
- CONTEXT.md's "Two maps — do not conflate" rule is updated: both maps are now
  Leaflet, but they still share no code or data — `createLombokMap` (app.js) is
  listings-only, `zoning.js` is zoning-only.
